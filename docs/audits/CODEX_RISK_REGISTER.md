# Codex Risk Register - Intellia237 Mobile

Date: 2026-06-18  
Scope: repository audit only. No production secrets are included.

| Severity | Count |
| --- | ---: |

| Elevated | 11 |
| Medium | 8 |

## Register

| ID | Severity | Area | Risk | Evidence | Impact | Recommended Action |
| --- | --- | --- | --- | --- | --- | --- |
| R-001 | Critical | Flutter build quality | App currently fails static analysis and widget test loading. | `flutter analyze` reports 5 errors in `lib/app/theme/app_theme.dart:97-98`; `flutter test` fails on the same missing `CupertinoPageTransitionsBuilder`. | Blocks reliable build/test gate and likely blocks release validation. | Fix import/API usage, then rerun `dart format --output=none --set-exit-if-changed .`, `flutter analyze`, and `flutter test`. |
| R-002 | Critical | Rewards and academic integrity | Quiz attempts and XP can be client-authored. | `lib/features/quiz/data/firestore_quiz_attempt_service.dart` increments XP with `attempt.xpAwarded`; `firestore.rules:286-287` allow students to create/update own attempts. | Users can fabricate XP, scores, progress, streaks, and leaderboards. | Move scoring, attempt persistence, XP, and badge updates to trusted backend Functions. Make client writes append-only or disallowed. |
| R-003 | Critical | AI architecture | Requested Gemini/Kira/Leo target is not implemented in tracked backend. | `functions/src/config/env.ts` uses `GLM_*`; README references Gemini; no tracked `llm-service/`; tutors are hard-coded Edunova personas. | The product cannot meet the stated target without architecture change. | Decide provider architecture, implement Gemini `gemini-3.5-flash` or approved fallback, and migrate tutor personas/prompts/assets deliberately. |
| R-004 | Critical | Secret handling | Functions log partial API-key material. | `functions/src/llm/llmClient.ts` logs key length and first/last characters when `GLM_API_KEY` is present. | Credential fragments can land in logs and incident exports. | Remove key logging entirely. Add structured redaction policy and logging tests. |
| R-005 | Elevated | AI cost control | Callable LLM Functions have no visible quota, subscription, App Check, per-user throttling, or spend guard. | `askTutor`, `generateQuiz`, and `generateSummary` require auth but no rate/quota layer is observed. | Abuse can generate high LLM cost or degrade service. | Add App Check, per-user rate limits, daily quotas, budget guards, and server-side abuse telemetry. |
| R-006 | Elevated | Firestore trust boundary | Progress, streaks, settings, and conversations are broadly user-writable. | `firestore.rules` allows owner writes for `progress`, `lessonProgress`, `streaks`, `settings`, and `ai_conversations`. | Academic state and personalization can become unreliable. | Classify each collection as user preference, user event, or authoritative academic record; restrict authoritative records to backend writes. |
| R-007 | Elevated | Product completeness | Several visible portals are mock/demo-backed. | `MockAdminRepository`, `MockParentRepository`, `MockTeacherRepository`, and `DemoStudentHomeRepository` are wired through providers. | Users see non-production data and workflows. | Replace mock repositories with Firestore/API implementations or gate these screens as beta/demo. |
| R-008 | Elevated | Firebase initialization | App does not pass generated Firebase options in bootstrap. | `lib/bootstrap.dart` calls `Firebase.initializeApp()` while `lib/firebase_options.dart` documents `DefaultFirebaseOptions.currentPlatform`. | Web/desktop and multi-project startup can fail or use unexpected config. | Use explicit platform options where required and test startup on Android, iOS, web, and emulator. |
| R-009 | Elevated | Dependency security | Functions dependency tree has known vulnerabilities. | `npm install` reports 83 vulnerabilities, including 12 high and 2 critical. | Security exposure and blocked compliance review. | Run `npm audit`, triage exploitable paths, upgrade dependencies, and document accepted residual risks. |
| R-010 | Elevated | Runtime mismatch | Local install uses Node 22 while package requires Node 20. | `npm install` warns current Node `v22.15.1`, required Node `20`. | Local and deployed behavior can diverge. | Standardize Node 20 via `.nvmrc`, Volta, CI, and Firebase Functions runtime configuration. |
| R-011 | Elevated | AI retrieval quality | Tutor context query is broad and not filtered by class level. | `functions/src/services/askTutorUseCase.ts` uses `collectionGroup("lessons")`; comment notes class-level filtering issue. | AI can answer from wrong class/course context. | Store denormalized searchable metadata or query a curated retrieval index scoped by class, subject, and user permissions. |
| R-012 | Elevated | Input abuse | Tutor history text and persona fields are insufficiently bounded. | `functions/src/utils/validation.ts` caps history item count but not each history text; tutor fields are strings without max lengths. | Prompt injection, large payloads, and cost spikes. | Add per-field max lengths, role allowlist, persona server lookup, and prompt-injection hardening. |
| R-013 | Elevated | Release governance | No CI workflow was found. | `.github` directory is absent. | Quality gates rely on manual local runs. | Add CI for Flutter format/analyze/test, Functions install/test/build, Firebase rules tests, and JSON/schema checks. |
| R-014 | Elevated | Firebase rules coverage | No Firestore/Storage rules tests were found. | Test inventory contains one Flutter test and two Functions test files only. | Security regressions can ship silently. | Add emulator rules tests for every role and sensitive collection. |
| R-015 | Elevated | Query scalability | Firestore queries lack visible pagination/index coverage in several feature repositories. | Quiz and learn repositories fetch class/subject/course data without clear pagination; `firestore.indexes.json` only covers generated content. | Performance and missing-index failures as data grows. | Define query contracts, add composite indexes, pagination, and query tests with representative data volume. |
| R-016 | Medium | Maintainability | Several screens exceed 500 to 1000 lines. | Largest files include tutor selection, student home, registration flow, lesson viewer, and quiz result screens. | Higher regression risk and slower feature work. | Split into smaller widgets/controllers after launch blockers are fixed. |
| R-017 | Medium | Asset size | Several image assets are large. | `assets/icons/edunova.png` is about 2 MB; onboarding and tutor images are roughly 0.6 to 1.0 MB each. | Larger app size and slower cold loads. | Optimize/compress assets and verify splash/icon variants. |
| R-018 | Medium | Documentation drift | README describes architecture not present in tracked code. | README references Gemini/FastAPI microservice and `llm-service/`; tracked code uses Functions plus GLM client. | New contributors can deploy or configure the wrong system. | Update docs only after architecture decision, with environment and deployment truth matching code. |
| R-019 | Medium | Monetization and quotas | No payments, subscription, or credit ledger implementation was found. | No Stripe/payment/subscription collections or services were observed. | Cannot safely monetize AI usage or premium tutoring without new backend work. | Design credit ledger, entitlement model, billing provider integration, and abuse controls. |
| R-020 | Medium | Admin operations | Client-side admin content CRUD is broad. | `admin_content_providers.dart` writes classes, subjects, chapters, lessons, and quizzes directly from Flutter. | Validation, moderation, and audit trails are limited. | Move high-impact admin writes to Functions with schema validation and audit logs. |
| R-021 | Medium | iOS release readiness | iOS bundle ID is set, but signing team was not observed. | `Runner.xcodeproj` uses `com.edunova.app`; no `DEVELOPMENT_TEAM` value was found in reviewed project settings. | Archive/signing can fail on release machine. | Decide bundle identity and configure Apple team/profiles outside this audit. |
| R-022 | Medium | Navigation/product growth | No deep links were observed. | Android manifest has launcher activity only; iOS plist shows no URL scheme/deep-link config. | Sharing lessons, quiz links, auth callbacks, and campaigns are limited. | Add deep-link strategy after final package/brand identity decision. |
| R-023 | Medium | Observability | No visible structured metrics for AI, costs, latency, or learning events. | Functions log errors/info but no metrics pipeline or dashboards are defined. | Operations and incident response will be weak. | Add structured event taxonomy, error budgets, latency/cost metrics, and alerting. |
| R-024 | Low | Branding consistency | Edunova naming remains throughout project. | `pubspec.yaml`, Android label, iOS display name, assets, README, and prompts use Edunova. | User-facing mismatch with Intellia237 target. | Treat as planned migration, not a search-and-replace, because package/Firebase/store identity decisions are involved. |
| R-025 | Low | Bootstrap resilience | Onboarding preference and Firebase startup use timeouts/fallbacks. | `lib/bootstrap.dart` continues after initialization failures. | Failures can be masked and surface later as UX issues. | Add startup diagnostics and user-safe degraded states. |
| R-026 | Low | Avatar moderation | Users can write own avatars directly to Storage. | `storage.rules` allows owner writes under `avatars/{uid}/...`. | Inappropriate content or oversized uploads without server processing. | Add upload constraints, moderation, and image processing if avatars are public. |
| R-027 | Low | Manual scripts | Utility scripts rely on local credentials/emulators. | `scripts/smoke_test.py`, `seed_sample_course.py`, and `ingest_drive_to_storage.py` exist without CI integration. | Useful checks may not run consistently. | Document required env and move safe smoke coverage into CI/emulator jobs. |
